# AI-Cascading Solo Development (ACSD)

## 1. Философия

В традиционной разработке код является источником истины, а документация отстаёт.  
В разработке с AI документация — это **единственный источник истины**, а код — её реализация.

Методология ACSD строится на пяти принципах:

### 1.1 Декларативная истина
Документация описывает целевое состояние системы (Target State).  
Не содержит статусов задач, истории изменений или TODO. Разница между документацией и кодом — это список невыполненной работы.

### 1.2 Каскадная валидация
Артефакты проекта образуют иерархию:  
Vision → Architecture → Module Architecture → Interface → Tests → Code

**Каждый уровень обязан соответствовать всем вышестоящим уровням.** Любое противоречие считается архитектурной ошибкой и устраняется через исправление кода или пересмотр документации.

### 1.3 Исполняемая спецификация
Тесты становятся **исполняемой формой интерфейсной спецификации**:  
Interface → Tests → Code

Это интеграция принципов TDD в ACSD. Тесты генерируются на основе `interface.md` и служат проверкой соответствия реализации спецификации.

### 1.4 Управление вниманием
Для соло-разработчика и AI минимизируется необходимый контекст:  
vision.md, ai_context.md, capsule.md, module/interface.md, module/architecture.md, tests  
Остальная часть проекта игнорируется.

### 1.5 Single Source per Concern
Каждый архитектурный аспект имеет **единственный документ-источник**. Другие документы только ссылаются на него и не дублируют информацию. Это предотвращает рассинхронизацию и архитектурный дрейф, особенно на больших проектах.

## 2. Структура проекта


# Структура проекта ACSD

```
project_root/
├── docs/
│   ├── vision.md           # Конституция проекта: мета-цель, стек, глобальные ограничения
│   ├── architecture.md     # Глобальная архитектура системы
│   ├── ai_context.md       # Консолидированный контекст для AI
│   └── adr/                # Architectural Decision Records
│
└── modules/
    └── module_name/
        ├── capsule.md            # Самодостаточный контекст модуля для AI
        ├── docs/
        │   ├── architecture.md   # Внутренняя архитектура модуля
        │   └── interface.md      # Публичный контракт (API, DTO, инварианты)
        ├── tests/
        │   ├── contract/         # Contract tests для проверки соответствия интерфейсу
        │   └── integration/      # Integration tests для взаимодействия модулей
        └── src/                  # Реализация модуля
```

## 3. Уровни артефактов

### Уровень 1 — vision.md (Конституция)
- Meta Goal, Tech Stack, Global Constraints, Core Principles
- Ограничение: 50-100 строк
- Роль: мгновенная перезагрузка контекста проекта

### Уровень 2 — architecture.md (Глобальная архитектура)
- Структура модулей, взаимодействие сервисов, глобальная модель данных, инфраструктура, архитектурные паттерны

### Уровень 3 — module/docs/architecture.md
- Внутренняя структура модуля: компоненты, алгоритмы, библиотеки, бизнес-логика
- Не нарушает правила верхних уровней

### Уровень 4 — module/docs/interface.md
- Контракт модуля: API endpoints, структуры данных, DTO, события, публичные классы
- Не содержит логики
- Структура для AI:
  1. Обзор модуля
  2. Владелец данных
  3. Публичный API
  4. Схемы данных
  5. Модель ошибок
  6. События (опционально)
  7. Инварианты

### Уровень 5 — tests/
- Contract tests проверяют соответствие реализации интерфейсу
- Integration tests проверяют взаимодействие модулей
- Генерируются на основе `interface.md`, не редактируются вручную

### Уровень 6 — src/
- Финальная реализация
- Код, противоречащий документации или тестам, считается ошибкой

### Уровень 7 — capsule.md (Module Capsule)
- Самодостаточный контекст модуля
- Структура:
  1. Идентификация модуля
  2. Ответственности
  3. Владеемые данные
  4. Зависимости
  5. Публичный интерфейс
  6. Внутренние компоненты
  7. Инварианты
- Служит для уменьшения контекста AI и предотвращения архитектурного дрейфа

## 4. Принципы документации

- **Декларативность:** описывает только целевое состояние  
- **Отсутствие состояния:** нет TODO, задач, roadmap, истории  
- **Architectural Decision Records (ADR):** фиксируют решение, альтернативы, причины выбора  
- **Single Source per Concern:** каждый архитектурный аспект имеет один документ-источник

---

## 5. Процесс разработки

1. **Изменение спецификации:** разработчик меняет `interface.md`, `architecture.md`, `vision.md`
2. **Генерация тестов:** AI генерирует contract и integration tests
3. **Gap Analysis:** AI сравнивает тесты и код → выявляет Missing Implementation
4. **Генерация реализации:** AI пишет код до прохождения всех тестов
5. **Валидация каскада:** AI проверяет соответствие всех уровней:  
Код → Тесты → Интерфейс → Архитектура модуля → Архитектура → Vision
6. **Рефакторинг:** оптимизация кода без нарушения тестов и интерфейсов

---

## 6. Роль AI

- **Спецификатор:** читает документацию  
- **Генератор тестов:** создаёт исполняемую спецификацию  
- **Генератор кода:** генерирует реализацию  
- **Валидатор архитектуры:** проверяет каскадную целостность  

---

## 7. Особенности и рекомендации

- Все данные о модуле должны быть в **capsule.md** для AI  
- `interface.md` максимально структурирован, машинно-читаемый: API, схемы, ошибки, инварианты  
- **Правила реализации:**
  - Точная реализация интерфейса
  - Не придумывать новые поля
  - Не изменять схемы данных
  - Не добавлять не задокументированные эндпоинты
- `ai_context.md` содержит агрегированный контекст проекта для AI  
- Single Source per Concern предотвращает архитектурный дрейф и рассинхронизацию на больших проектах

---

## 8. Итоговая модель ACSD


Человек → Спецификация (vision, architecture, capsule, interface)
AI → Генерация тестов и кода
Тесты → Проверка соответствия


Разработчик управляет архитектурой и спецификацией, AI реализует код, а тесты гарантируют соответствие спецификации.
