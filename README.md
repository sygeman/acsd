# AI-Cascading Solo Development (ACSD)

## 1. Философия

В традиционной разработке **код является источником истины**, а документация отстаёт.
В разработке с AI **документация является единственным источником истины**.

Код рассматривается как **исполнение спецификации**.

Методология ACSD строится на четырёх принципах:

### 1.1 Декларативная истина

Документация описывает **целевое состояние системы (Target State)**.

Документация:

- не содержит статусов задач
- не содержит истории изменений
- не содержит TODO

Документация отвечает только на вопрос:

> Как система должна быть устроена.

Разница между документацией и кодом — это **объём невыполненной работы**.

---

### 1.2 Каскадная валидация

Артефакты проекта образуют иерархию.

Каждый уровень обязан соответствовать уровню выше.

```
Vision
↓
Architecture
↓
Module Architecture
↓
Interface
↓
Tests
↓
Code
```

Нижележащий уровень **не может противоречить** родительскому.

Если противоречие обнаружено:

- код считается ошибкой реализации
- или спецификация пересматривается

---

### 1.3 Исполняемая спецификация

Спецификация должна быть **проверяемой автоматически**.

Для этого между интерфейсом и кодом вводится слой **тестов**.

```
Interface → Tests → Code
```

Тесты являются **исполняемой формой интерфейсной спецификации**.

---

### 1.4 Управление вниманием

Соло-разработчик и AI имеют ограниченный контекст.

Методология минимизирует необходимый объём контекста.

Для работы над модулем требуется только:

```
vision.md
module/interface.md
module/architecture.md
tests
```

Остальная часть проекта игнорируется.

# 3. Уровни артефактов

## Уровень 1 — vision.md (Конституция)

Краткое описание фундаментальных принципов проекта.

Ограничение: **50-100 строк**.

Содержит:

- Meta Goal — цель системы
- Tech Stack — фиксированный стек
- Global Constraints — запрещённые практики
- Core Principles — ключевые архитектурные правила

Пример ограничений:

```
No ORM
No synchronous I/O
PostgreSQL only
Stateless services
```

Роль:
**мгновенная перезагрузка контекста разработки**.

---

## Уровень 2 — architecture.md (Глобальная архитектура)

Описание архитектуры всей системы.

Содержит:

- структуру модулей
- взаимодействие сервисов
- глобальную модель данных
- инфраструктуру
- архитектурные паттерны

Файл может быть большим.

---

## Уровень 3 — module/docs/architecture.md

Описание внутренней архитектуры модуля.

Содержит:

- компоненты
- алгоритмы
- используемые библиотеки
- бизнес-логику
- внутренние модели данных

Ограничение:

Модуль **не может нарушать правила** верхних уровней.

---

## Уровень 4 — module/docs/interface.md

Контракт модуля.

Содержит:

- API endpoints
- структуры данных
- DTO
- события
- публичные классы
- форматы входов и выходов

Файл **не содержит логики**.

Роль:

Создание **контекстного пузыря**, позволяющего разрабатывать модуль изолированно.

---

## Уровень 5 — tests/

Тесты являются **исполняемой спецификацией интерфейса**.

Основной тип тестов:

### Contract tests

Проверяют соответствие реализации интерфейсу.

```
input → output
```

Пример:

```
given valid email
when POST /users
then status = 201
```

### Integration tests

Проверяют взаимодействие модулей.

---

## Уровень 6 — src/

Код реализации.

Правило:

Если код противоречит документации или тестам —
**код считается ошибкой**.

---

# 4. Принципы документации

## 4.1 Декларативность

Неправильно:

```
Нужно добавить авторизацию
```

Правильно:

```
Система обеспечивает авторизацию через JWT
```

---

## 4.2 Отсутствие состояния

Документация не содержит:

- TODO
- статусы
- roadmap
- задачи

Она описывает **идеальную систему**.

---

## 4.3 Архитектурные решения (ADR)

Каталог:

```
docs/adr/
```

Каждый файл фиксирует:

- проблему
- выбранное решение
- альтернативы
- причины выбора

ADR объясняет **почему архитектура устроена так**.

---

# 5. Процесс разработки

Разработка происходит циклически.

## Шаг 1 — Изменение спецификации

Разработчик изменяет документацию:

```
interface.md
architecture.md
vision.md
```

Это изменение **целевого состояния системы**.

---

## Шаг 2 — Генерация тестов

AI генерирует тесты на основе интерфейса.

```
interface.md → tests
```

Тесты фиксируют ожидаемое поведение системы.

---

## Шаг 3 — Gap Analysis

AI сравнивает:

```
tests ↔ code
```

Определяется список:

```
Missing Implementation
```

---

## Шаг 4 — Генерация реализации

AI генерирует код до тех пор, пока:

```
all tests pass
```

---

## Шаг 5 — Валидация каскада

AI проверяет соответствие:

```
code → tests
tests → interface
interface → architecture
architecture → vision
```

Если найдено противоречие —
либо код исправляется, либо обновляется спецификация.

---

## Шаг 6 — Рефакторинг

После прохождения тестов AI может оптимизировать код.

Условия:

- тесты должны оставаться зелёными
- интерфейс не изменяется

---

# 6. Роль AI в ACSD

AI выполняет четыре функции:

### Spec Interpreter

Интерпретирует документацию.

---

### Test Generator

Создаёт исполняемую спецификацию.

---

### Code Generator

Генерирует реализацию.

---

### Architecture Validator

Проверяет каскадную целостность системы.

---

# 7. Основная идея методологии

ACSD меняет модель разработки.

Традиционная разработка:

```
Human → Code → Docs
```

ACSD:

```
Human → Specification
AI → Implementation
Tests → Verification
```

Разработчик управляет **архитектурой и спецификацией**,
AI реализует **конкретную реализацию**.

Таким образом разработка становится процессом
**сведения реализации к спецификации**.
