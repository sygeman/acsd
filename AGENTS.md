# ACSD v2.3 Specification

## Core Principle
Documentation is the single source of truth. Code is an implementation detail. Tests are the arbiter.
If Code != Docs -> Code is wrong. If Docs != Reality -> Docs are wrong (requires ADR).

## Critical Nuances
Boundary: L0-L3 (Vision..Interface) edited by Human; L4-L5 (Tests..Code) generated by AI.
ADR: The only valid path to change Architecture/Vision based on Code insights.
State: .acsd/ is ephemeral cache. Git commits are permanent truth.

## Mandatory Frontmatter Rule
ALL cascade documents MUST start with YAML frontmatter block:
---
template_version: "1.0"
file_type: "<vision|architecture|data_model|capsule|interface|adr>"
cascade_level: <0-5>
required_fields: [field1, field2, ...]
cascade_children: ["relative/path/*.md", ...]
---
Scripts validate frontmatter before any processing.
Missing or invalid frontmatter = ERROR status (not CONFLICT).
Frontmatter is machine-readable; content is human+AI readable.

## Cascade Hierarchy (Strict Top-Down)
L0: acsd/vision.md (Purpose, Scope, Actors, Invariants)
L1: acsd/architecture/ (Components, Boundaries, Tech Stack) - file or folder with index.md
L1: acsd/data_model.md (Entities, Relations, Ownership)
L2: **/<module-root>/acsd-module/architecture/ (Module specific structure) - file or folder with index.md
L2: **/<module-root>/acsd-module/capsule.md (Domain concepts, Operations)
L3: **/<module-root>/acsd-module/interface.md (Public API, DTOs, Errors)
L4: **/<module-root>/tests/contract/*.test.ts (Generated from Interface)
L5: **/<module-root>/src/*.ts (Generated from Tests, stack-specific paths)

## Document Structure Rule
Any cascade level (except L0 vision, L3 interface) may be:
- Single file: architecture.md
- OR folder: architecture/ with index.md as entry point

Folder structure example:
  acsd/architecture/
    ├── index.md           # Required: entry point, links to all other files
    ├── components.md      # Optional: detailed component descriptions
    └── boundaries.md      # Optional: interaction rules

index.md must:
- List all files in the folder
- Define relationships between them
- Be the only file referenced by parent/child cascade links
- Contain frontmatter that aggregates required_fields from subfiles

## Module Structure
<module-root>/
  ├── acsd-module/        # L2-L3: ACSD artifacts
  │   ├── architecture/   # OR architecture.md (file or folder with index)
  │   ├── capsule.md
  │   └── interface.md
  ├── src/                # L5: Implementation (stack-specific)
  └── tests/              # L4: Contract tests (stack-specific)

CLI discovers modules by finding all **/acsd-module/ folders.
Module root = parent directory of acsd-module/.
Code and tests live alongside acsd-module/, not inside it.

## File States
OK: Spec matches Code and Tests pass.
PENDING: File needs processing.
ERROR: Syntax error, generation failure, or invalid frontmatter.
CONFLICT: Semantic mismatch (Spec != Code). Requires human decision.
EXCEPTION: Temporary approved drift (must have expiry).

## Conflict Resolution Protocol
1. Detection: Tests pass but contradict Spec, or Code cannot implement Spec.
2. Classification: AI marks file as CONFLICT.
3. Proposal: AI suggests: (A) Fix Code, (B) Update Spec, (C) Create ADR.
4. Decision: Human chooses option.
5. Application: Changes applied, cascade re-validated.
Rule: Code can propose Spec change via ADR, but never ignore Spec silently.

## Invariants (Hard Rules)
1. Docs are source of truth.
2. Contract tests must pass for Code to be valid.
3. CONFLICT status requires explicit human resolution.
4. System state is tied to Git commit hash.
5. Dependencies stored in SQLite (.acsd/acsd.db).
6. AI handles semantics; Scripts handle syntax/paths/git.
7. Module boundary marked by acsd-module/ folder (discoverable via glob).
8. Implementation code lives alongside acsd-module/, not inside it.
9. Stack-specific conventions for code/tests paths documented in ADR.
10. Folder-based docs must have index.md as single entry point.
11. ALL markdown files MUST have valid YAML frontmatter as first block.

## Git Workflow
Standard:
1. git checkout -b feat/name
2. Edit vision/interface docs
3. acsd validate --scope module:name
4. Resolve CONFLICTs manually
5. Commit valid state
6. PR -> CI runs acsd validate --full -> Merge

Hotfix:
1. git checkout -b hotfix/name
2. Fix code (bypass cascade if critical)
3. Commit with [EXCEPTION] tag
4. Create task to update spec later

## CLI Reference
acsd init                   # Init DB and structure
acsd discover               # Find all acsd-module folders
acsd validate [--full]      # Run cascade validation
acsd status                 # Show current state
acsd drift                  # List CONFLICTs
acsd adr create <title>     # Start decision record

## LLM Instructions
- Validate: Check frontmatter exists, required_fields present, parent links valid.
- Generate: Derive children from parents strictly, include frontmatter in output.
- Conflict: Flag mismatches, propose diffs, do not auto-merge up.
- Context: Always read parent files before processing child.
- Discover: Find module root from acsd-module/ parent directory.
- Folder docs: Read index.md first, then linked files. Never reference subfiles directly in cascade.
- Frontmatter: Always preserve and update frontmatter when editing content.

## Anti-Patterns
- Committing without validation
- Ignoring CONFLICT status
- Letting AI write vision.md without human review
- Merging to main without full cascade check
- Creating EXCEPTIONs without expiry date
- Placing implementation code inside acsd-module/
- Assuming code paths without checking stack ADR
- Referencing architecture subfiles directly (always use index.md)
- Folder without index.md entry point
- Markdown file without frontmatter block
- Editing content without updating frontmatter required_fields
